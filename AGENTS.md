# AGENTS.md — condash (Electron)

Electron build of condash. As of 2026-04-27 this **is** the canonical condash — it lives at `vcoeur/condash` after the swap that moved the Tauri lineage to `vcoeur/condash-tauri`. The Tauri build remains buildable at that repo for as long as bug fixes are warranted.

## Stack

- **Main**: Node, TypeScript, **bundled with esbuild** (`scripts/build-electron.mjs`) to a single CJS file at `dist-electron/main/index.js`. Imports are inlined; ESM-only deps (chokidar 4, future libs) bundle to CJS transparently. Native modules (electron, node-pty, fsevents) stay external — they have to load from `node_modules` so `electron-rebuild` can reach them.
- **Preload**: same bundler, output at `dist-electron/preload/index.js`. Stays CJS because `webPreferences.sandbox: true` + ESM preload don't mix on Electron.
- **Typecheck**: `tsc -p tsconfig.main.json --noEmit` (and the renderer twin). `tsc` no longer emits — esbuild owns emission, tsc owns type-checking.
- **Renderer**: Solid + Solid signals, Vite (with `vite-plugin-solid`), plain CSS files + CSS variables, `src/renderer/main.tsx`.
- **Renderer boot chunk is on a diet** — heavy deps are **lazy-split** out of the eager entry chunk and loaded on first use, not at first paint: **xterm + its 9 addons** (`src/renderer/xterm-mount.ts`, dynamic-imported on first terminal open; the `liveTerms` registry + `refreshAllXtermThemes` live in the leaf `xterm-registry.ts` so `use-theme` can repaint terminals without pulling xterm); **markdown-it + highlight.js** (`src/renderer/markdown.ts` builds the engine lazily — `renderMarkdown`/`highlightCode` are async; the note/help/html modals `await` them via `createResource`); the **plan-block MDX parser** (`src/shared/plan-blocks/parse-mdx.ts`, micromark/mdast + zod — the mdx-modal dynamic-imports it on first `.mdx` open, and dompurify + jsdiff load the same way inside the block components); mermaid + CodeMirror were already split. Keep new heavy deps off the eager path the same way. (Eager chunk: 1.5 MB → ~0.4 MB at v4.32.0.)
- **CLI cold start is on a diet too** — every `condash <noun>` is a fresh Node process that loads the single CJS bundle, so module-*init* cost (not parse) is paid per invocation. Two disciplines keep the fixed baseline ~halved (≈120 ms → ≈70 ms on the fast verbs): (1) `src/cli/index.ts` **lazy-dispatches** — its `dispatch()` switch `await import('./commands/<noun>')`s the one handler it routes to instead of statically importing all twelve, so a call only evaluates the module graph it needs (`simple-git` loads only for `repos`/`worktrees`; the other handlers' deps never load for `help`/`config`/`knowledge`/…). **Never re-add a static handler import at the top of `cli/index.ts`.** (2) The **config read path is zod-free** — `migrateRawSettings` + the `DEFAULT_*` path constants live in `src/main/config-migrate.ts` (no `zod` import), so `resolveConception` → `settings.ts` / `effective-config.ts` / `conception-paths.ts` never construct the ~313 KB of strict schemas in `config-schema.ts`; `config get` / `set` lazy-`import()` the schemas only on the write-time validation path. Keep new read-path code importing from `config-migrate.ts`, not `config-schema.ts`.
- **Shared types**: `src/shared/types/` — plain serialisable objects, ISO strings, no methods. Split into domain modules (`common`, `project`, `layout`, `settings`, `git`, `terminal`, `logs`, `task-runs`, `search`, `tree`) re-exported from a thin barrel at `src/shared/types/index.ts`, so `@shared/types` (and `../shared/types`) imports resolve unchanged. Add a new type to its domain module; reach for the barrel only when importing several domains at once. Renderer view-state (`WorkingSurface`/`LeftView`/`LayoutState`) lives in `layout.ts`, kept under `shared/` because the layout is persisted through the settings IPC contract.
- **Packaging**: electron-builder, single `BrowserWindow`, all modals/panes are in-renderer overlays.
- **Modal shell**: every simple centered modal renders through `<Modal>` (`src/renderer/modal.tsx`), which owns the backdrop, centered panel, `.modal-head` (title / path / `headExtra` / close button), dialog ARIA, and the two shared behaviours — Esc-to-close (`useModalEscHandler`) and drag-out-safe backdrop dismissal (`createBackdropClose` from `modal-helpers.ts`). Pass per-modal layout via the `class` width-tier (`modal-w-sm/md/lg`) and bespoke head content via `headLeading` / `headExtra`. The richer surfaces keep their own layout because their Esc contract is non-trivial: the **note** modal (dirty-guard + in-modal back-stack via `modal-router.ts`), the **settings** modal (unsaved-edits Esc gate), and the **project-preview** popover (lets inline edit inputs swallow Esc). Confirmations go through the single `ConfirmModal` (no per-case copy). Singleton menu/keyboard overlays (search, settings, new-project, about, quit-confirm, shortcuts, help) share one `activeModal` discriminated-union signal in `hooks/use-modals.ts` (mutual exclusion in one place); the payload-carrying, router-coupled surfaces (note `modal`, `previewPath`, pdf/html/image) keep their own signals.
- **Watcher**: single global chokidar rooted at `<conception>/`, debounced 250 ms. Also keeps the in-memory **search index** fresh (`src/main/search/index-cache.ts`): the four Markdown sources (projects incl. notes, knowledge, resources, skills) are precomputed in RAM at conception-open and queried without re-reading the tree; `applyIndexFsEvent` updates one file per FS event. **Logs are not indexed** — too large, rarely searched — so they're disk-scanned, and only when in scope. The renderer's default **All** filter forwards the four indexed scopes (`ALL_SCOPES` in `search-modal.tsx`), not "everything", so a default query is index-only (~45 ms); the log disk-scan runs only under the explicit **Logs** filter. `search()` falls back to the on-disk scan only while the index is still building. See `docs/explanation/internals.md#search-index`. The watcher likewise invalidates the **`parseReadme` memo** (`src/main/parse-cache.ts`, keyed on path + mtime) on a README's change/unlink, so `listProjects`/`getProject` re-parse only changed files instead of the whole tree on every reload. Event→reload classification is a **pure, unit-tested** module (`src/main/watch-classify.ts`): an in-project file (note, `local/` asset) patches just that one card, an `index.md` regen is ignored, a project dir add/remove reloads only the project list, and only a genuinely unrecognised path falls to `unknown` (the whole-dashboard fan-out) — see `docs/explanation/internals.md` §6.

### Adding a new main-process dep

1. `npm install <pkg>`. ESM-only is fine — esbuild bundles it as CJS.
2. If the dep is **native** (has a `binding.gyp` or ships prebuilt `.node` files — e.g. `node-pty`), add it to `scripts/shared/externals.mjs` (`SHARED_EXTERNALS`) so esbuild leaves it alone for both the Electron and CLI builds, and wire `electron-rebuild` if needed. Per-target externals (Electron-only or CLI-only) go in the per-script `EXTERNAL` array instead.
3. Pure-JS deps need no further work.

### Dependencies

#### `@xterm/*` beta pinning

The xterm v6 line only ships as `beta.NNN` prereleases on npm — there is no stable v6 yet. Every `@xterm/*` package therefore carries an **exact** pin (no caret), and all packages in the family must stay on the **same** `beta.NNN` minor so their internal peer ranges resolve. When bumping xterm:

- Pick the latest `beta.NNN` of `@xterm/xterm` and align every other `@xterm/*` package to the same `beta.NNN` in one commit.
- One known exception: `@xterm/addon-webgl@0.20.0-beta.NNN` declares a peer of `@xterm/xterm@^6.1.0-beta.(NNN+1)`, so the addon ends up one tag *behind* the rest of the family. This is upstream metadata, not a bug — accept the off-by-one until upstream tags catch up.
- `@xterm/addon-fit` and `@xterm/headless` still publish stable releases (`^0.11.0`, `^6.0.0`) and are excluded from the beta-pin sweep.

## Locked decisions

The load-bearing design choices — UI framework (Solid + Solid signals), styling (plain CSS files + CSS variables), IPC contract shape (single typed `CondashApi` in `src/shared/api.ts`), window architecture (single `BrowserWindow`, in-renderer overlays), data shape (plain serialisable objects in `src/shared/types/`), file-watching protocol (single global chokidar rooted at `<conception>/`, 250 ms debounce), config format (JSON: per-machine `settings.json` + per-conception `.condash/settings.json`, gitignored by default), and build tool (esbuild for main/preload, Vite for renderer) — are captured in this file and in [`docs/explanation/internals.md`](docs/explanation/internals.md). Treat them as locked: changing any of them is a PR with a dated rationale in the commit message, not a silent in-flight edit.

## Dev ports

- **5600** — Vite dev server.
- **5601** — Vite preview.
- Both `strictPort: true`. Defined in `vite.config.ts`. The numbers are picked from a 56xx block to stay clear of common defaults (3000, 5173, 5432, 8000, 8080) and of sibling vcoeur apps that bind their own host ports during dev.

When changing the dev port, update **every** file:

- `vite.config.ts` — `server.port`, `preview.port`.
- `Makefile` — `DEV_PORT`, `PREVIEW_PORT`.
- `src/main/index.ts` — `DEV_URL`.
- `src/renderer/index.html` — CSP `connect-src` (HMR websocket).
- `package.json` — `dev:electron` `wait-on tcp:<port>`.
- This file.

### Tests are headless by default — never open windows on the dev's screen

A Playwright run launches the **real Electron app**, which on a Wayland desktop renders to the live compositor (or XWayland) and **steals focus mid-run**. The headless guarantee lives in `npm run test` → `scripts/run-playwright.mjs`, which wraps the whole run in a throwaway **Xvfb** display with `WAYLAND_DISPLAY` dropped and the **X11 Ozone backend pinned** (`--ozone-platform=headless` is not usable — Electron can't attach Playwright to it). A *direct* `npx playwright test` that skips the wrapper is **aborted by the globalSetup guard** (`tests/fixtures/headless-guard.ts`) before any window opens.

- **Always run the suite via `npm run test` / `make test`** (or `npm run test -- <spec>` for one file). Never `npx playwright test` / `xvfb-run npx playwright test` directly — the first aborts, the second double-wraps.
- **After touching `src/`, use `make test` (or `npm run build` first).** The fixture launches the *built* app (`main` → `dist-electron/`), and bare `npm run test` has no build step — so it silently exercises the previous build and reports green on code that is not the code you changed. Only `npm run test -- <spec>` on an already-current build is safe.
- A visible run (to watch it) is the explicit opt-in `CONDASH_TEST_HEADED=1` (`make test-visible`).

## Configuration

- **Per-machine**: `${XDG_CONFIG_HOME:-~/.config}/condash/settings.json`. Owned by Electron's `app.getPath('userData')`, derived from `package.json`'s `name` field. JSON only. Sole owner of `lastConceptionPath` + `recentConceptionPaths` (cap 5); also carries global defaults for every workspace key.
- **Per-conception**: `<conception>/.condash/settings.json` (canonical). The whole `.condash/` directory is fully gitignored by default — settings included — so it carries per-host state with no commit-leak risk. Two legacy paths still read as fallbacks: `<conception>/condash.json` (old canonical) and `<conception>/configuration.json` (legacy²). Writes always target `.condash/settings.json`. The auto-migrator (`src/main/condash-dir-migrate.ts`) copies legacy content on first open, tombstones the source file, and appends `.condash/` to `.gitignore` when the conception is a git repo. `condash config migrate` exposes the same operation explicitly.
- **Per-tab memory containment (Linux + systemd + cgroup v2)** — each terminal tab's pty spawns inside its own transient `systemd-run --user --scope` with `MemoryHigh`/`MemoryMax`/`MemorySwapMax` (`src/main/tab-scope.ts`, wired into `terminals.ts` `spawnTerminal` + the headless `task-scheduler.ts` spawn), so a runaway agent tab trips its **own** cgroup OOM and dies alone instead of a global OOM taking the whole app down (the 2026-07-05 crash: a ~10 GB cline team exhausted RAM+swap and the collateral disposed condash's renderer). Capability is probed with a throwaway scope; a **success is cached** but a **transient failure is re-probed** on the next spawn (so a momentary glitch never silently disables containment for the whole session — the escape hatch behind the 2026-07-05 recurrence), and a capable-host miss logs once; unsupported hosts spawn directly. The per-tab scope only binds children spawned through the tab path, so condash also caps its **own** app scope at startup (`capOwnAppScope` → `systemctl --user set-property MemoryMax/MemorySwapMax` on the derived `app-*condash*.scope`; config `terminal.memory.appScope {enabled?,max?,swapMax?}`, default RAM−3G / 2G): a **backstop** so a child that escapes per-tab scoping (a probe miss, a non-tab helper, a stale pre-cap instance) trips condash's cgroup OOM instead of a global one that kills the whole GNOME session (the exact 2026-07-05 recurrence). `--scope` (not `--service`) keeps the child in condash's session so it inherits the node-pty pty/cwd/env — verified needed for interactive TUIs. Config: `terminal.memory {enabled?,high?,max?,swapMax?,appScope?}` (global, default **enabled** — high 6G / max 8G / swapMax 2G; `appScope` default enabled), all editable in **Settings → Terminal → Memory containment**. Main samples each scoped tab's cgroup `memory.current` on a 2.5 s unref'd timer and rides it on the existing `termSessions` snapshot (`TermSession.memBytes`/`memMaxBytes`, sampled only for scoped tabs — an unscoped pid resolves to condash's own cgroup); the tab strip renders a per-tab meter that `.warn`s at ≥ 80 % of cap (`terminal-pane/column.tsx`).
- **Death attribution + perf counters** — `pty.onExit` captures node-pty's **`signal`** as well as the exit code and reads the tab cgroup's `memory.events` **by a path cached at spawn, never by pid — and never before the child has migrated into its scope**. Both halves cost a release. Reading at exit fails 100 % of the time (node-pty emits `exit` only after `waitpid` has reaped the child *and* the pty socket has closed, so `/proc/<pid>` is already gone), silently reducing every OOM to a bare SIGKILL, and a recycled pid would read a foreign cgroup and manufacture a spurious verdict. But reading *immediately* after `pty.spawn` is just as wrong: `systemd-run` execs before it has asked the user manager to create the unit and move into it, so the read returns **condash's own app scope** — every tab then caches one foreign path, reports the whole app's memory as its own, and derives its death verdict from the app's counters (shipped in v4.96.0; the tell is every Performance row showing an identical figure and moving in lockstep). Hence the scope is **named** (`--unit=condash-term-<id>.scope`, `condash-task-` for scheduled runs — the two id generators both mint `t-<hex>`, so the kind prefix is what stops a collision failing the spawn) and `resolveScopeCgroup` polls until the path ends in that unit name. Confirm by unit name, not by "changed from the parent": a sibling tab's scope also differs from the parent. No resolution → **no figures**, never a guessed path. The exit-time read still races `--collect`, so the session keeps the **last two** periodic samples and falls back to them; `src/main/term-death.ts` (pure, no electron/node-pty import, unit-tested like `terminal-flow.ts`) derives a six-way verdict that rides the `termExit` payload, the `termSessions` snapshot, and the log footer. Two rules are load-bearing: (1) `memory.events` counters are **cumulative for the cgroup's life**, so a verdict must diff against the baseline the 2.5 s sampler keeps — testing `high > 0` labels every later kill on a long-lived tab a pressure kill; (2) the **cap** case is checked before the **pressure** case, because a cap hit drives reclaim on the way up and raises `high` too. The two OOM verdicts are genuinely different failures: a cap hit raises the cgroup's own `oom_kill`, while an external killer reacting to PSI pressure (systemd-oomd) does **not** — and the latter is what kills tabs in the field, at `MemoryHigh`, never reaching `MemoryMax`. An **abnormal** exit now keeps its tab row (verdict badge + **Restart**, a main-side verb since the `Session` record holds `cwd`/`cmd`/`side`); only a clean `exit 0` auto-closes, because auto-closing an abnormal exit destroyed the only evidence. `src/main/perf-log.ts` accumulates main-thread counters (event-loop delay via `perf_hooks monitorEventLoopDelay`, OSC-scan and logger-parse times as **separate** spans, grid renders, flow batches/pauses/watchdogs) behind `terminal.perf.enabled` (**off by default**) and flushes JSONL to `.condash/perf/` **on the existing 2.5 s sampler tick — never its own timer**; `TerminalFlow` reports through an optional observer rather than importing it, keeping that module dependency-free. Surfaced by the **Performance** left-band pane (`src/renderer/panes/perf-view.tsx`), which also shows per-tab memory **growth rate** and a **throttled** state from the always-on sampler — both absent before, and the reason a tab climbing to its cap inside one sampling window gave no warning. `scripts/perf-load.mjs` stages an N-tab flood for the logging on/off A/B **into a throwaway user-data dir + conception under `/tmp`, asserted in the main process before any load** — it drove the user's real app state until 2026-07-22, leaving disk logging permanently toggled and evicting real prior-day transcripts through the log janitor's oldest-first sweep. Isolation removes that class outright and removes the need for a restore path that a Ctrl-C would skip. It launches with **`CONDASH_FORCE_PROD=1`** so the unpackaged run loads the built `dist/index.html` instead of the Vite dev URL: without it `isDev` (`index.ts:187`) is true, the renderer dies on `ERR_CONNECTION_REFUSED` against a dev server nobody started, and the run still exits 0 — measured 2026-07-22, that dead renderer understated `gridRenderMs` by 26 %, understated the logging A/B's loop-p99 delta by **9×**, and reported `pauses: 0` where the real path pauses 4 times (flow control cannot engage when nothing consumes). In forced-prod mode the dev `app.setPath('userData', …)` redirect no longer runs, so `--user-data-dir` carries the isolation on its own and `CONDASH_DEV_USER_DATA_DIR` stays only as an inert backstop against the prod force being dropped; the runtime assertion reads the **live** path rather than trusting either mechanism. **Every precondition is asserted, never assumed** — the harness measures other software, so an unverified assumption inside it becomes plausible false data rather than a crash: the preflight checks **both** build halves (a main-only check passed while `dist/index.html` was absent), the renderer must have **mounted** (`#root` has children — the preload runs on the chrome-error page too, so `window.condash` being present proves nothing), and a `--trace-gc` run counts the GC records it actually captured, warning loudly at zero rather than throwing — by then the run's other counters are already valid, so only a GC conclusion is off-limits (V8 writes those to **stdout**, so capturing only stderr left `gc.log` holding Electron noise and **zero** GC records for every run before 2026-07-22 — a documented capability that had never worked). The harness also refuses >12 tabs or a run exceeding available memory without `--force`, since per-tab caps are per tab and the documented failure is whole-machine pressure. **`--profile` picks the load's SHAPE, and it decides which half of the logger the run can see at all** — default `flood`, so every pre-existing invocation is unchanged. The logger renders out of a headless xterm capped at `scrollback + ROWS` = **5050 rows** and flushes every **5 s**, and since v4.97.1 the render reuses the previous flush's frozen prefix — so what matters is new rows per flush *against 5050*. `flood` at its default 512k emits ONE 10924-char line per chunk (base64 of 8192 raw bytes, `tr`-joined) which wraps at 200 cols to **55 rows**, × 64 chunks/s = 17 600 rows per flush = **3.5× full buffer turnover**: nothing is ever retained, the frozen prefix is always empty, and a retention-dependent optimisation is invisible *by construction* — measured 2026-07-23, the v4.97.1 incremental grid render scores **exactly zero** here and 1.4–1.6× in the realistic regime. `realistic` (default rate **16k**, not the flood's — inheriting it would give 6.8× turnover, *worse* than the flood) emits 80–119-char lines, one grid row each, ~806 rows per flush = **16 %** of the buffer, in bursts with idle gaps, from a fixed seed so both `--ab` arms see byte-identical input. It also cuts the fork rate ~100× (lines are literals, `printf` is a builtin, only the per-burst `sleep` forks) — the long-documented caveat that flood figures are upper bounds contaminated by ~2000 process creations/s. **The rate is the crux, not the line shape**: short lines at the flood rate still land 34 400 rows per flush. A realistic-profile run shorter than ~60 s measures the buffer warming up (~31 s to saturate), not the steady state; the harness prints its regime and that warm-up time at startup. Two verdict guards keep the death record honest: an **intentional** stop (condash's own SIGKILL pipeline) never yields an OOM verdict, and counters whose window does not bracket the death (`bracketsDeath: false`, the `--collect` fallback) ride along for the footer but never promote one. Measured 2026-07-22: the exit-time read **wins** its race with `--collect` — the cgroup is readable at `onExit` and gone ~50 ms later, for clean exit, slow exit, and SIGKILL alike — so the fallback is the rare path, not the common one. `monitorEventLoopDelay` records the gap between its **own** timer firings, not the excess over the expected gap, so a raw reading has a floor equal to its `resolution` — at `resolution: 10` an idle process measures p99 ≈ 10.3 ms, and a known 100 ms block reads 106.5 ms. Shipped raw in v4.96.0, that put a fixed ~10 ms (61 % of a frame budget) on the pane's headline "main loop p99" for an idle app — the symptom under investigation, at a plausible magnitude, i.e. an instrument positioned to confirm its own hypothesis. `loopDelayMs` subtracts the resolution and clamps at 0; subtract rather than lower it, since a 1 ms resolution fires a timer 1000×/s on the very thread being measured. **Three traps, all locked by tests:** (1) `setTerminalPrefs` **replaces** the whole `terminal` block despite reading like a patch — read-merge-write or a toggle silently wipes shell/shortcuts/logging/memory (`ipc/perf-set-enabled.test.ts`). (2) `layoutSchema.leftView` is built with `z.enum(LEFT_VIEWS)` from `shared/types/layout.ts`, **never hand-listed** — the hand-written union went stale when a view was added, and because `updateLayout` spreads the persisted layout into every write, one unlisted value made *every* later layout save throw, not just the new pane's (`ipc/settings-set-layout.test.ts` iterates `LEFT_VIEWS`). (3) Any per-tab figure folded into the sampler's `changed` flag needs its own broadcast **quantum** — the growth rate is a fresh integer every tick, so an exact compare rebroadcasts the whole session snapshot every 2.5 s and undoes the T5 idle-churn fix for every user with memory scoping on, whether or not they use the pane (`rateChanged`, `terminals.test.ts`).
- **Main-process crash resilience** — top-level `process.on('uncaughtException'|'unhandledRejection')` guards (`src/main/index.ts`, registered after CLI dispatch so GUI-only) log instead of letting a stray throw in a pty/watcher/scheduler callback kill the app; `app.on('render-process-gone')` reloads a disposed renderer window (an OOM/GPU cascade otherwise leaves a wedged blank window while the pty `onData`/`onExit` handlers keep sending to the dead frame — the observed "Render frame was disposed" spam); `app.on('child-process-gone')` logs helper (network/GPU) crashes. **Every** main→renderer push funnels through the shared `safeSend` helper (`src/main/safe-send.ts`: `isDestroyed()` + `isCrashed()` + try/catch, returning whether the payload reached a live frame) — the pty data/exit path plus the tree + repo watchers, the task scheduler, the dashboard engine, and the menu — replacing the bare `webContents.send` / `isDestroyed()`-only guards that a disposed-but-not-destroyed frame slipped past (a throw there aborts the window loop and lands in the global guard as stderr spam).
- **Per-conception logs**: `<conception>/.condash/logs/YYYY/MM/DD/HHMMSS-<sid>.txt`. **One single plain-text file per pty spawn** since v2.27.0 — no sidecar, no compression. Metadata travels inside the `.txt` as two `# condash: {...}` JSON lines: a header line at the top (always present, carries `{sid, side, repo?, cwd, cmd, argv, started, kind}`, where `kind` ∈ `transcript` | `grid` records which body this flush wrote) and a footer line at the bottom (only after `exit()`, carries `{finished, exitCode}` plus `death` — the derived verdict, so a kill is diagnosable from the log alone; `exitCode` cannot distinguish one, since node-pty reports 0 for a SIGKILLed process). `cat foo.txt` shows the metadata too. Gitignored alongside `settings.json`. **Opt-in** since 2.25.0 — `terminal.logging.enabled` defaults to `false`; flip it in `Settings → Terminal → Logging` to start recording. The pref is read once at session-start (writer construction). The writer (`src/main/terminal-logger.ts`) pipes pty bytes into a **lazily-constructed** headless xterm (`@xterm/headless` — allocated only on the first grid byte, so a logging-off spawn or a pure OSC-transcript session never pays for one), every 5 s renders the buffer row-by-row via `IBufferLine.translateToString(true)` (plain text — no SGR, no CSI, no cursor-forward), composes header + body (+ footer if exited), and writes the `.txt`. The periodic flush keeps its cost proportional to NEW output, not retained size: a **transcript** flush appends only the lines added since the last write (`open(path,'a')` with the delta), driven by a per-flush line watermark (`bodyCursor`) into the extractor plus a compact byte-length + short tail sample of what's on disk, rather than re-joining the whole (multi-MB) transcript and prefix-comparing the whole file; a **grid** flush skips the whole-buffer `translateToString` render entirely when no new bytes reached the term since the last write, and when it does render, `GridBodyRenderer` re-translates only the rows that can still have changed (this bounds the **row walk** only — joining the rows and rewriting the file are both still O(retained size), and after this change they are the whole cost: measured at production geometry the walk fell from ~21 ms to ~12.5 ms per render, of which ~12.8 ms is the `join` alone) — rows above the viewport (`[0, baseY)`) are out of the cursor's reach forever, so their rendered text is cached and reused. Scrollback eviction renumbers every row, so that cache is anchored to an xterm **marker** (which xterm shifts per evicted line and disposes once its own line is evicted) rather than to an absolute index; the marker is pinned at `baseY - 1`, **never** at `baseY`, because `CSI L` on the viewport's top row inserts at index `baseY` and drags a marker pinned there one row into the viewport, and a later eviction cancels the sign of that slide so the arithmetic guard stops rejecting it — the result is a body silently off by one row. A buffer swap (alternate screen, or `RIS`, which installs brand new buffer objects) strands the marker, so the renderer subscribes to `onBufferChange` itself, in its constructor, rather than trusting a caller to wire it. The *file* write is still a full repaint rewrite — a grid body isn't append-shaped — and the reused rows are retained in memory (~1 MB at the 5000-line default, against the several MB the headless term's own buffer already holds). Any inconsistency (a byte-cap trim dropped written lines, the header `kind` flipped, the on-disk length/tail no longer matches, a prior write error) and every durable **exit/close** flush fall back to the atomic tmp→(fsync)→rename full rewrite, which re-establishes the file and the compact bookkeeping exactly — only that watermark is retained in memory now, never the whole file text (the G9/T3/T8 write-amplification + retention fix). Flushes are serialised through `flushChain` so `close()` can await every pending write before disposing the xterm. `in` keystrokes are *not* captured — the pty echoes them back through `out`. Session size is bounded twice: the grid body by xterm scrollback (`terminal.logging.scrollback`, default 5000 lines), and the OSC transcript body — which scrollback doesn't touch — by both a line cap and a byte cap (`MAX_TRANSCRIPT_LINES` + `MAX_TRANSCRIPT_BYTES` in `src/main/osc-transcript.ts`, oldest lines dropped past either — the byte cap matters because one message line can reach a few MB); no rotation. Each pty chunk is OSC-scanned **once** in main: `terminals.ts` scans it for the always-on dashboard extractor (`session.transcript`) and hands the logger the stripped text + decoded frames (`feedCapturingFrames` → the logger replays them via `applyDecodedFrame`), so the logger's own transcript extractor is populated without re-scanning the same bytes; the dashboard reads its tail via `renderTail(maxChars)` (a backward walk over the retained lines, byte-identical to `render().slice(-maxChars)`) instead of re-joining the whole transcript (T4/T8). Janitor (`src/main/terminal-logger-janitor.ts`) (1) evicts day-directories past `terminal.logging.retentionDays` (default 14), (2) evicts oldest-first while over `terminal.logging.maxDirMb` (default 500); runs at startup + every 24 h. The Logs working surface (`Cmd+Shift+L`) is now a **collapsible card grid grouped by date** (`src/renderer/panes/logs.tsx`) — each session is a card showing time · repo · cmd · exit · size in a responsive grid sized by `cardMinWidth.logs`. Grouping: the last 7 days each get a collapsible `<details>` day group (today is rendered as a static, always-expanded block — no toggle); older days fold into collapsible per-month `<details>` groups, each with a light day sub-header. Collapse state is native `<details>` (not persisted), so today is always expanded on open. Partition is pure-render off the `logsListDays` list (string day-compare against a 7-day cutoff); no IPC change. Clicking a card opens `LogsViewerModal` (`src/renderer/logs-modal.tsx`): a full-overlay viewer with a virtualised plain-text transcript (fixed-row-height windowing over `lines.length * --logs-row-height` via an absolutely-positioned slice), a search box that pre-computes per-line hit indices and scrolls the active hit into view, and `Esc` to close. Long lines horizontal-scroll (no wrap) so the virtualizer's fixed-row-height math holds. Logs are also a source in the global search modal — `src/main/search/walk.ts:collectLogFiles` walks `<root>/YYYY/MM/DD/*.txt`; `src/main/search/match.ts` strips the `# condash:` header / footer lines via `splitContent` (exported from `src/main/logs-format.ts` — the pure-parsing module shared between the renderer-side `ipc/logs.ts`, the writer in `terminal-logger.ts`, and the CLI's search command; carved out so the CLI bundle's import graph doesn't touch `electron` or `@xterm/headless`) before substring matching, so snippets don't quote the JSON. Activating a log hit posts a `LogsOpenRequest` that opens the modal directly. Legacy `.jsonl` files from condash ≤ 2.22 and `.txt.gz` from 2.23–2.26 are ignored by the viewer — the janitor evicts them by age. The same logs have a read-only CLI surface since v4.12.0: the `logs` noun (`src/cli/commands/logs.ts`) with `days` / `list` / `read` / `tail`, backed by the CLI-safe query core `src/main/logs-query.ts` (enumerate / filter by date·mtime·repo·active / slice by head·tail·line-range·byte-cursor) — carved out CLI-safe for the same reason `logs-format.ts` was. On a conflict the owned-file value wins, but the reconciliation is value-typed: an **object** key is **deep-merged** into the owning file (owned leaf wins, so sub-keys that were legitimately split across the two files under the old override model — the load-bearing case is `terminal.{screenshot_dir,…}` in `settings.json` + `terminal.logging` in `.condash/settings.json`, whose wholesale drop silently disabled terminal logging) survive; a scalar / array key is dropped wholesale. Documented in `docs/reference/config.md`; tested in `effective-config.test.ts`.
- **Field-naming convention (frozen)**: the user-facing config surface mixes two casings and existing keys are **never renamed** (a rename is a breaking settings migration). A guard test in `config-schema.test.ts` rejects any top-level key that is neither pure snake_case nor pure camelCase (`$`-prefixed meta-keys like `$schema_doc` excepted).

Shape-level migrations ahead of the strict zod parse live in `migrateRawSettings`, which sits in the **zod-free `config-migrate.ts`** (so the CLI read path can run it without constructing the schemas — see the CLI cold-start note in **## Stack**); `config-schema.ts` re-exports it for the write/GUI canonicalisers.
- **Agents** are a flat `{ id, label, command }` list under the top-level `agents` key (`agentSchema` in `src/main/config-schema.ts`, part of `globalOnlyFields` so it lives only in the per-machine `settings.json` — a personal key, identical across conceptions, and rejected in a `.condash/settings.json`). Each is a terminal launcher: picking it opens a new tab running `command`. There is **no** per-file store, no harness model, and no secret store — a launched command inherits the terminal's ambient environment, so provider/model wiring lives in the command (often a `~/bin` wrapper script). `src/main/agents.ts` is a thin reader: `listAgents(conceptionPath)` returns the config's `agents`, skipping rows with a blank `id` or `command` (half-filled Settings-editor placeholders); IPC is the single `listAgents` verb (`src/main/ipc/agents.ts`). The shared type is `Agent` in `src/shared/types/terminal.ts`. An optional `promptFlags: true` marks the command as understanding agedum's `--prompt` / `--run` flags, so tasks/actions seed the prompt in argv instead of typing it (see **Tasks**). Agents are edited via the **Launchers** section of the Settings modal (`src/renderer/settings-modal-parts/sections-agents.tsx`) — a single personal-scope section (no per-conception copy; the old dual-tab + inheritance machinery was removed in the scope-partition revamp), a vertical list of `{label, command, id}` cards (+ a "Seed prompt via flags" checkbox) with add / remove / move-up-down (no DnD — Wayland-Ozone HTML5 DnD is broken). There is no dedicated Agents pane or working surface. Blank-row survival mirrors repositories: `agentSchema` fields all accept `''`, and `buildSavePayload`'s `compactAgents` bypass keeps a `{id:'',label:'',command:''}` row clear of `pruneEmpty`. The tab-strip spawn dropdown (`terminal-pane/column.tsx`) lists `New shell` + each agent by `label`; selecting one spawns via `TermSpawnRequest.command = agent.command` (the existing command path in `terminals.ts` — no agent resolution). Migration: `migrateRawSettings()` still drops legacy `terminal.launchers` / `launcher_command` and renames an action-template `launcher` → `agent` (now an agent `id`). (History: a richer per-file harness subsystem — `<conception>/agents/<slug>.json` + `agents/.env` + claude/kimi/opencode/agentsconf spawn builders + an Agents pane — was removed in favour of this `{id,label,command}` model.)
- **Tasks** are reusable, parameterized agent prompts. A task references an agent (by `id`) + a markdown prompt with fillable `{markers}`, stored as one directory each at `<conception>/tasks/<slug>/`: `task.json` (`name` / `agent` (the agent's `id`) / `submit`) + a hand-editable `prompt.md` (prose stays in markdown, config in JSON — the conception "never mix shapes" rule). Loaded by `src/main/tasks.ts` (ENOENT-tolerant `listTasks` — `agentPresent` checks the `id` against the `agents` settings list — slug-validated writes via `isValidSlugTail`, idempotent `deleteTask`, `writeTask` rejects a rename onto an existing slug); IPC `listTasks` / `readTask` / `writeTask` / `deleteTask` (`src/main/ipc/tasks.ts`). The Tasks pane (`src/renderer/panes/tasks.tsx`) is a **left-band view** (`LeftView` adds `'tasks'`, ordered Projects · Tasks · Deliverables) — a responsive card grid sized by `cardMinWidth.tasks` (each card is click-to-edit and carries a single **Run…** button that `stopPropagation`s) + a **run popup** (pickers + prefilled fields + live preview → Run) + an **editor popup** with Save/Cancel + a `ConfirmModal`-confirmed Delete. Marker grammar + the `{APP_*}` / `{PROJECT_*}` reserved-token context builders live in `src/shared/tasks.ts` (`extractMarkers`, `appContext`, `projectTokenContext`); `substitute` (`src/shared/action-template.ts`) honours `{KEY:default}` as a fallback. Run goes through `terminal-bridge.ts`'s `runTask(agentId, text, taskName, opts)` → shared `runAgentTask(agent, text, submit, title, taskContext, mode)` (also used by agent-bound project/new-project actions, which pass no `mode` and so default to `interactive`). For an opaque agent it spawns a fresh tab running the agent's command (cwd = conception root), then types the substituted prompt and submits when `submit` is true (keystroke-injected via the pty after a settle delay; `mode` is moot — the keystroke path is interactive only). For an agent with `promptFlags`, it instead folds the prompt into argv per the run **mode** — `<command> --run '<text>'` (one-shot: runs then exits) or `<command> --prompt '<text>'` (interactive: seeded, session stays open) — and spawns that, typing nothing (single-quoted via `shellSingleQuote`). A task's mode comes from `taskConfig[slug].runMode` (`'interactive' | 'oneshot'`, default interactive), set in the editor and overridable per run in the run popup; the headless scheduler (`task-scheduler.ts`, `resolveRunMode`) reads the same field so a `oneshot` scheduled run exits cleanly rather than being killed at its `timeout`. The editor's agent select stores the agent `id` and shows its `label`; a dangling `agent` id (removed from settings) is flagged on the card (`agentPresent`) and disables Run.
- **Dashboard (live tab summaries)** is an opt-in feature that periodically summarizes the open terminal tabs by POSTing directly to an OpenAI-compatible LLM endpoint (DeepSeek by default) — a cheap **card** model that pre-processes each tab into facts + a draft title, plus a richer **writer** model that composes each tab's published title + one-sentence subtitle from those facts. The engine (`src/main/dashboard/engine.ts`, armed/disarmed alongside the task scheduler in `index.ts`) ticks every 15 s but schedules **per tab, not on one global cycle**: each open tab carries its own next-refresh time (`nextDueAt`, seeded from `intervalSec` + a fixed per-sid jitter ≤ one tick so tabs that would come due together desync) and is refreshed independently when its own clock comes due; the engine ETA is the earliest due tab, and the per-card **Update** button (`refreshTab`) resets that one tab's clock. Each tick rebuilds its open-tab roster from `dashboardRoster()` (`terminals.ts`: the user's `side: 'my'` terminal tabs only — Code-pane **Run** sessions are `side: 'code'` dev servers, not agent tabs, so they neither inflate the tab tallies nor render as idle cards; `tabsContext()` stays both-sides for the task `{TABS}` var), prunes summaries + scheduler bookkeeping (clock/baseline/jitter) for closed tabs every tick (no LLM, no key) so a card/status never outlives its tab, gates each due tab on the same byte-growth check the scheduler uses (`tabsBytes()` in `terminals.ts`), reads each grown tab's recent output via `tabRecentText()` (ANSI-stripped + CR-resolved in `dashboard/clean-text.ts`, sourced from the in-memory buffer so it works whether or not on-disk logging is on), and summarizes the due tabs **in parallel** (`Promise.all`) with stateless, tool-free completions (`dashboard/summarizer.ts`: `runCompletion` does a direct `fetch` POST to `<baseUrl>/chat/completions` — no SDK, no fs/shell access). The **card** model (`config.model`, over a wide `cardInputChars` window) pre-processes the raw output into a per-tab `{title, contextLines, currentAction, state, activity}` — `state` is the coarse `working`/`awaiting`/`idle`/`error` the cards colour on, `activity` a finer work stage (`implementing`/`designing`/`reviewing`/`making-pr`/`documenting`/`testing`/`debugging`/`researching`/`awaiting`/`idle`), and its `title` is a **draft** the writer polishes (and falls back to). Local, **no-LLM provenance** (`dashboard/provenance.ts`, `deriveProvenance`) then maps the tab's `repo` → app `#handle` (reuses `findRepoEntry`), derives the worktree/branch dir name when the tab `cwd` sits under `worktrees_path`, and matches that branch against the conception projects index (`findProjectReadmes` + `branchToDir`) to list related projects (`{slug, title}`). The **writer** model (`config.writerModel`) is fed those card facts + the provenance and composes the per-tab published **title** (3–7 words) + one-sentence **subtitle** — the work's context + purpose — from the distilled facts (it never sees the raw output, so it polishes rather than re-grounds), falling back to the card model's draft title when its reply omits one so the card's most prominent field never depends on the pricier call (`writeCard` + `CARD_WRITER_SYSTEM_PROMPT`); there is **no** cross-tab `{globalWork, overview, events}` synthesis any more (the old `synthesizeOverview` / `OverviewResult` / `OVERVIEW_SYSTEM_PROMPT` were removed, and `DashboardState` dropped `globalWork` / `overview`). Secret redaction (`redactSecrets`) still scrubs every captured-content field before it is POSTed — the tab command/cwd/recent output for the card call, and the draft title/action/context + provenance names for the writer call (the `state`/`activity` enums are model-derived and carry none). Reasoning is controlled per call via DeepSeek's `thinking:{type:"disabled"}` switch (`buildCompletionBody`) — **off by default for both tiers** (`cardReasoning` / `writerReasoning`): a model bake-off found reasoning-on adds 4–10× latency with no title gain on the card tier and returns an empty reply on a non-trivial fraction of writer calls (unacceptable now the writer owns the title). Sampling temperature is **per-model** (`temperatureForModel`) — `0` (greedy, stable across the every-cycle re-summarize) everywhere except Moonshot/kimi models, which reject any temperature but `1` (a hardcoded `0` would 400 every cycle on them). Because the gate only re-summarizes a tab when its bytes grow, a tab that finishes and falls silent would otherwise stay frozen on its last `working` badge (the transition to idle — output stopping — is exactly what the gate reads as "nothing to do"); so each tick first runs a local, LLM-free **idle-decay** (`decayStaleWorkingTabs` in `engine.ts`) that flips any `working` tab with no byte growth for `DECAY_INTERVALS`× the interval to `idle`, leaving `awaiting` and `error` untouched (their quiet is real). The card prompt carries the complementary fixes at the source: a coding agent that printed a complete reply and returned to an empty input box is `idle`, not `working`; an agent merely offering suggestions or asking an open-ended "what next?" is `idle`, not `awaiting` (reserve `awaiting` for a concrete blocking prompt); and a title is never derived from an example/suggestion/placeholder line. A custom `baseUrl` (any OpenAI-compatible endpoint — DeepSeek's gateway, an opencode-go / Zen server, …) is POSTed to directly; with no `baseUrl` it falls back to `https://api.deepseek.com`. A named `User-Agent` is always sent — some gateways Cloudflare-403 the default `fetch` UA. State (per-tab summaries — each now carrying `subtitle`, `activity`, and the optional `app` / `worktree` / `projects` provenance — plus a rolling event history + a `lastError` surfaced when a cycle's API calls fail) persists at `<conception>/.condash/dashboard/state.json` (`dashboard/state.ts`) and pushes to the renderer over the `dashboard-tab-summaries` / `dashboard-state` event channels, driving the LLM tab title (`displayName` in `terminal-pane/types.ts`), a hover popover (`terminal-pane/column.tsx`), and the **Dashboard** body in the bottom band (`renderer/panes/dashboard.tsx`). That body is a Direction-B layout: a **minimal top status line** (product label, an on/off dot for enabled+keyed, live per-state tab tallies, the last-update time) whose **hover popover** shows the full summarizer config (provider, the card + writer models with their reasoning flags, the interval, the activity gate, and whether the API key / base URL are set — secrets never shown, only set/missing), any last-cycle error below it, then one **card per open tab** with a state health dot, the per-card **Update** button (`dashboardRefreshTab` IPC → `refreshTab()` in `engine.ts`, a single-tab summarize that bypasses the interval + activity gate) + relative age, the title with an **activity-stage pill**, a provenance breadcrumb (`#app › worktree › project`), the one-sentence subtitle, and a recent-actions list; a tab with no summary yet keeps a fallback card drawn from its command/cwd, and a card being recomputed shows a small "summarizing" pulse. There is **no** cross-tab overview block and **no** old left rail / 3-segment status strip / "Open tabs · N" header. The body is selected by an always-first, fixed **Dashboard** pseudo-tab in the left tab strip (`terminal-pane/column.tsx`; not draggable/renamable/closable). Activating any real terminal tab switches the band back to the terminal body (`onShowTerminalBand` in `main.tsx`, non-toggling); the Dashboard tab itself toggles the band via `bottomView` state (also `Ctrl+Shift+D` / View → Show Dashboard). The xterm hosts are CSS-hidden (`terminal-pane.css` `.dashboard-active`), not unmounted, so terminals survive and refit on return. Config is the `dashboard` block (`config-schema.ts`), a **global-only** key edited in the single `settings-modal-parts/sections-dashboard.tsx` section under **Personal · this machine** — the whole block (`enabled`, `apiKey`, `baseUrl`, `model`, `writerModel`, `cardReasoning`, `writerReasoning`, `cardInputChars`, `intervalSec`, `gateOnActivity`) is editable there. No per-conception copy and no merge, since `getEffectiveConceptionConfig` in `effective-config.ts` is a plain spread of the two disjoint schemas. The section has a **Test connection** button (`dashboardTestConnection` IPC → `testDashboardConnection` in the summarizer). Defaults + the 30–300 s interval clamp + the `cardInputChars` floor live in `dashboard/config.ts`. The summarizer makes plain `fetch` calls — **no SDK** — so nothing heavy enters the renderer/startup path or the CLI/main bundle (the former pi coding-agent dependency was dropped). It installs a no-op `worker_threads.markAsUncloneable` shim (`ensureUndiciElectronShim`) before the first `fetch`: Electron 33's Node 20.18 doesn't export that symbol and the undici behind global `fetch` reads it with no fallback, so a fetch would otherwise throw `webidl.util.markAsUncloneable is not a function`. Off by default; enabling transmits recent terminal output (secret-redacted via `redactSecrets`) to the configured endpoint (DeepSeek by default, or whatever `baseUrl` points at — privacy note in `docs/reference/config.md`).

## Sandbox: dev vs. production

- **Dev** (`make dev`) — `dev:electron` passes `--no-sandbox`. Avoids the per-worktree `sudo chown root chrome-sandbox` ritual. The dev window only loads `http://localhost:5600` and local `file://` URLs, so the threat surface is local-only.
- **Production** (`make package`) — `electron-builder` SUIDs the helper at install time on `.deb` and AppImage; the sandbox is on for end users.

If you want the sandbox on during dev anyway: drop `--no-sandbox` from `dev:electron` in `package.json` and run once per worktree:

```bash
sudo chown root node_modules/electron/dist/chrome-sandbox
sudo chmod 4755 node_modules/electron/dist/chrome-sandbox
```

macOS and Windows are unaffected.

## Conventions

- Run `make format` after every code change.
- All UI must work without per-OS CSS branches (Electron + Chromium = same renderer everywhere).
- **CSS design system.** Every text `font-size` uses a `--text-*` px token (`--text-3xs` 10px … `--text-3xl` 24px) — no rem ladders, no half-pixel sizes, no raw px on text (icon glyphs are the only exemption and say so in a comment). There is **no** spacing scale: padding/gap/margin are raw px on purpose (the tokenised scales are type, radius, and the modal width tiers). Themes come from the registry in `src/shared/themes.ts`; each preset is one `[data-theme='<id>']` token block in `styles.css` and never re-types another theme's hex literals. Rules that only need dark-vs-light key on `[data-theme-kind='dark']` (stamped by `use-theme.ts` alongside `data-theme`), never on a preset id — that is what lets a new dark theme land without touching a single pane stylesheet. Define tokens before using them (`--danger`, `--text-dim`, `--preview-*` are real); never fake one with `var(--made-up, #literal)`. Shared shape primitives (`.section-header`, `.pill`) live in `src/renderer/primitives.css`. **Actions are one vocabulary** in `src/renderer/actions.css`, rendered by `<Button>` / `<IconButton>` / `<ActionBar>` (`actions.tsx`): call-sites name the role (`.btn--primary` filled-accent commit, `.btn--default` cancel/neutral, `.btn--ghost` text action, `.btn--danger` destructive; `.btn--sm` / `.btn--icon` modifiers) and the verb via `data-tone` (`open`/`add`→accent, `work`/`run`→`--col-running`, `stop`/`danger`→`--warn`) — never a per-pane button class. Footers use `.action-bar` (cancel left, primary right); toggles use `.seg`/`.seg-item`. Full spec: [internals invariant 11](docs/explanation/internals.md#css-design-system).
- **Component split threshold.** A renderer pane/modal `.tsx` past **~400 LOC** or holding more than one sub-component is split into a sibling `<name>-parts/` directory; `<name>.tsx` keeps the public component + orchestration and imports the rest. Fixed sub-file vocabulary: `data.ts` (pure logic + types + constants, dependency-free, with a `data.test.ts`), `icons.tsx` (pane-local glyphs), and one presentational `.tsx` per sub-component. A `-parts/` modal renders through the shared `<Modal>` shell. The threshold is a smell, not a gate — a genuinely single-responsibility long file (e.g. `index-tree.ts`) stays whole. Full spec: [internals invariant 12](docs/explanation/internals.md#component-parts).
- In-window drag must use pointer events (`pointerdown` / `pointermove` / `pointerup` + `setPointerCapture`), never HTML5 drag-and-drop — Chromium's Wayland Ozone backend (forced for crisp text on Wayland sessions) silently breaks HTML5 DnD. The Projects-pane status drag follows this; terminal-tab and settings-row reorder are still HTML5 DnD (broken on Wayland) until converted. See [internals invariant 8](docs/explanation/internals.md#8-in-window-drag-uses-pointer-events-not-html5-drag-and-drop).
- File-path conventions: every path crossing the IPC boundary is normalised to forward-slash form via `toPosix(p)` from `src/shared/path.ts`. Internal `fs` calls keep the native separator (use `path.join`); the renderer can then split on `/` without per-OS branches. Cross-OS shell wrapping (`bash -lc` / `cmd.exe /d /s /c` / `pwsh -Command`) lives in `src/main/terminals.ts`'s `wrapForShell`.
- Spawned-process PATH: every pty / child process condash spawns takes its PATH from `spawnEnv()` in `src/main/shell-env.ts`, which resolves the user's **login-shell PATH** once at boot (`$SHELL -lic`, cached, 5 s timeout) so GUI-launched condash (Wayland / macOS — no `~/.profile` sourcing) still finds user-installed CLIs (`opencode`, `~/bin` wrappers). PATH only; other vars keep inherited values so the env-hygiene scrub stays intact. Wired into `terminals.ts` (pty + `force_stop`) and `launchers.ts`; warmed in `index.ts` at `whenReady`.
- Renderer CSP carries `'wasm-unsafe-eval'` in `script-src` (`src/renderer/index.html`) for `@xterm/addon-image` — its Sixel decoder is WebAssembly, which Chromium refuses to compile under a bare `'self'`. The refusal throws inside xterm's synchronous write loop, blanking any terminal that receives an inline-image escape (e.g. the opencode TUI). `'wasm-unsafe-eval'` permits WASM compilation only, **not** JS `eval` / `new Function` — keep it; do not widen to `'unsafe-eval'`.
- IPC contract is the single typed `CondashApi` interface in `src/shared/api.ts`. Functions, not URLs. Promises for request/response; events only for chokidar push. The `invoke`/`handle` channel equals the `CondashApi` method name (the interface anchors both ends), but the **main → renderer push-event** channels (`tree-events`, `termData`, `menu-command`, …) have no such anchor, so they live as named constants in `src/shared/ipc-channels.ts` (`EVENT_CHANNELS`) — reference that map from both the `webContents.send(...)` (main) and `ipcRenderer.on(...)` (preload) ends rather than re-typing the literal. Verb-prefix convention: `read*`/`write*` for file-backed channels, `get*`/`set*` for small scalar prefs (the one documented exception is `getGlobalSettingsRaw`/`writeGlobalSettings`, flagged in `api.ts`).
- **IPC handlers validate renderer args through the shared decoders** in `src/main/ipc/utils.ts` (`requireNonEmptyString` / `requireBoolean` / `requireStringArray` / `requireOptionalStringArray` / `requireEnum`), not hand-rolled inline `if (typeof …) throw` guards. Each throws `<channel>: expected …` so the trust boundary reads uniformly. Add a decoder there rather than re-typing a guard. Every `ipcMain.handle` body additionally starts with `requireMainWindowSender(event)` (same module) — the sender must be the app's own top frame, never a `<webview>` guest. A handler that returns a typed value for a no-op case must return the well-formed empty shape (e.g. `search` → `emptySearchResults()` in `src/shared/types/search.ts`), never a bare `[]` — the declared `Promise<T>` is a contract.
- **Path-bounding** for any renderer-supplied path goes through `src/main/path-bounds.ts`, which builds every variant (`requirePathUnder` / `requirePathUnderWorkspace` / `requireReadableSkillPath`) on one realpath primitive (`isRealpathUnder` + `resolveUnderAnyRoot`). Apply the bound at the IPC handler (the choke point). The shell-out verbs `openPath` / `showInFolder` are bounded via `requireOpenablePath` in `ipc/system.ts` (workspace roots plus one exact-file exemption for the per-machine `settings.json`); the only handler deliberately left unbounded is `openInEditor` — the trust boundary is documented at its call site.
- **Plan documents are one parser everywhere.** `src/shared/plan-blocks/` (registry → zod schemas → `parsePlanMdx`) is the single code path behind the `.mdx` plan viewer (`src/renderer/mdx-modal.tsx`), the `condash mdx check`/`blocks` CLI verbs, and the shipped `/visual` skill's `blocks.md` (generated by `renderBlocksDoc`, drift-pinned in `mdx-references.test.ts` — regenerate with `condash mdx blocks`, never hand-edit). MDX is parsed as DATA (static-literal attributes via `literal-eval.ts`), never compiled or evaluated — the renderer CSP has no `unsafe-eval` and must not need it. Author-supplied wireframe/diagram HTML renders only through `mdx-modal-parts/sanitize.ts`.
- **The CLI is a thin shell over the `src/main` engine, not a fork.** Commands under `src/cli/commands/*` reuse main/shared modules rather than re-implementing logic: the `**Verified:**` stamp grammar (`src/main/knowledge-stamps.ts`), the stale-stamp scan (`src/main/audit/stale-verification.ts`, shared with the `stale-verification` audit check + the GUI audit pane), the knowledge-tree walkers (`src/main/search/walk.ts`), the shared `Project` shape + `compareByStatusThenSlug` status comparator (`src/shared/projects.ts`), dotted-path config addressing (`src/shared/dotted-path.ts`), and the global-settings writer (`src/main/settings.ts` `mutateSettingsJson`, queued with `updateSettings`). CLI scaffolding is also shared: verb-based nouns dispatch through `runNoun` and render help via `renderHelp` (`src/cli/help.ts`); flag extraction uses `takeStringFlag` / `takeIntFlag` / `takeBoolFlag` (`src/cli/parser.ts`). When you reach for logic in a command file, check whether main/shared already owns it.
- **`sync` is the conception's only committer.** The `sync` noun (`src/cli/commands/sync.ts` over `src/main/sync/`) exists because a checkout shared by parallel agent sessions has three ways to corrupt itself — the process-wide `.git/index`, the fan-in `index.md` files no session owns, and racing pushes — and all three dissolve if exactly one process commits. So: nothing else in condash may `git add` / `git commit` / `git push` in a *conception* (repo worktrees under `worktrees setup` are a different tree and unaffected), every write happens under the non-blocking `<git-dir>/condash-sync.lock`, and a rejected push is reported rather than repaired (`git pull --rebase` would rewrite the tree under a live session). For that "nothing else commits" claim to hold for the *whole* checkout, `sync` sweeps **every tracked, non-gitignored change**, not just the two trees — a `meta` group (`classifyPath` in `src/main/sync/group.ts`) that catches everything outside `projects/`/`knowledge/` (`AGENTS.md`, `.agents/**`, config files, `resources/`, `tasks/`, …) into one `meta: sync` commit. It's a catch-all, not an allowlist, and that is safe because `git status` already omits gitignored paths: the compiled per-harness views (`.claude/*`, `.pi/*`, `CLAUDE.md`, …) never reach the sweeper, so the only way to keep a file out of `sync` is to gitignore it — anything left tracked is a file that would otherwise be stranded with no committer. The one exception is a malformed path *inside* a tree (`projects/stray.md`), reported `unresolved` rather than swept. A held-back `meta` path never defers index regeneration (an index references items, never `AGENTS.md`). The quiet-period mtime filter is what makes `sync run` safe to fire on a timer; don't add a code path that commits a path it hasn't aged. The rule is only load-bearing if the *conception's* agents read it, so it also ships in the condash-owned head of `conception-template/AGENTS.md` (`### Committing`, guarded by a test in `files.install.test.ts`) and in the `projects` / `knowledge` skills — `projects/close.md` closes an item by writing files only: the sweeper detects the new `Closed.` timeline entry (`src/main/sync/close-milestone.ts`) and synthesizes the `Close <slug>. Outcome: …` milestone subject itself, leaving `sync commit` as a manual escape hatch for humans. Change one, change them together.
- **Dead-code / over-export hygiene is gated by knip** (`make deadcode`, configured in `knip.json`; runs in the light gate). Keep `export` off module-internal helpers — only export what another module (or a test) imports. The gate fails on dead source files, dead / unlisted / unresolved deps, and duplicate exports (so a re-added back-compat alias is caught); the pre-existing unused-export / unused-type backlog is graded `warn` (non-failing) pending a dedicated sweep. Adding a new dependency? It must be imported, or `knip.json`'s `ignoreDependencies` must record why (as `electron-updater` does — tracked but intentionally unimported).
- **README/note mutation** is split by concern under `src/main/`: `mutate-steps.ts` (`## Steps` checklist edits), `mutate-status.ts` (**Status** transitions + `## Timeline`), `write-config.ts` (generic note write + settings/config canonicalisation — the one writer that knows the zod config layer), over a shared text toolkit (`mutate-shared.ts`: `detectEol` + the per-file write queue). `mutate.ts` is now a thin re-export barrel, so existing `./mutate` imports resolve unchanged.
- Hard interaction-to-paint budget: **≤ 16 ms** for any user-initiated action. Optimistic UI is the default; rollback on IPC failure.

Older `.deb` assets stay reachable from each release's GitHub page; only the apt index is windowed.

The in-app Help menu reads files out of the asar via the allowlist in `src/main/help.ts`. The allowlist carries six short names — `welcome`, `quick-start`, `shortcuts`, `configuration`, `cli`, `why-markdown` — each mapping to `docs/help/<name>.md`. Separate menu items link out to `condash.vcoeur.com` and the issue tracker via `shell.openExternal`. There is no separate copy at the repo root — the migration shipped in `v2.0.6`.

The first-launch **welcome screen** lives in `src/renderer/welcome-screen.tsx` and renders inside `workspace-center` when the conception path is set, the projects list is empty, and the knowledge tree is empty (and the user hasn't dismissed it). Dismiss state persists at `welcome.dismissed` in `settings.json` via the `getWelcomeDismissed` / `setWelcomeDismissed` IPC verbs.

The **status bar** (the `src/renderer/main.tsx` header; widgets in `src/renderer/status-bar-indicators.tsx`) carries two live sync indicators. The **auto-sync** pill reflects the sweeper's state (synced / *N* to sync / syncing / failed / off) with a *Sync now* button (`autoSyncNow`) and a click-to-open recent-commits popover, backed by the read-only `syncStatusSnapshot` verb (`src/main/sync/status-snapshot.ts` — pending-file count + unpushed count + recent commits; disjoint from the sweeper, it never takes the sync lock). The **shipped-skills** pill reflects whether `condash skills install` is in sync (`skillsSyncStatus` → `src/main/skills-sync-status.ts`, comparing the bundled shipped source at `<app>/conception-template/.agents/skills` against `<conception>/.agents/skills/` + the `.condash-skills.json` manifest); when files are missing/outdated it surfaces an *Install* action that runs `condash skills install` in a fresh terminal tab via the bridge's `runShellCommand`. Both read on a poll + on the `auto-sync-status` push.

The Code pane's **branch filter** (top-of-pane popover) shows every worktree by default; ticking branches narrows each card to "primary + ticked branches". Unticking everything returns to the default "show all" view. The primary worktree row (the checkout under `workspace_path`) is always rendered with a subtly tinted background. Hydration + persistence go through `createBranchFilterStore` in `src/renderer/branch-filter-store.ts` and the `getSelectedBranches` / `setSelectedBranches` IPC verbs (settings.json field: `selectedBranches`). Filter logic is the pure `filterWorktrees` in `src/renderer/panes/code-parts/data.ts`.

When changing app behaviour, update the matching `docs/` file in the same commit. The schema in `src/main/config-schema.ts` and the public reference (`docs/reference/config.md`) must agree on every release.
